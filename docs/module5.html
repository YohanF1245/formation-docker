Introduction au Stockage Docker

Objectifs Pédagogiques

Dans cette section, vous allez apprendre :

  • Les concepts fondamentaux du stockage Docker
  • Les différents types de stockage disponibles
  • Comment choisir le bon type de stockage
  • Les bonnes pratiques de gestion du stockage

Concepts de Base du Stockage

Système de Fichiers en Couches

Docker utilise un système de fichiers en couches qui permet :

  • La réutilisation efficace des couches entre conteneurs
  • L'optimisation de l'espace disque
  • Des démarrages rapides des conteneurs
  • La gestion des modifications de manière isolée

Types de Stockage

  • Volumes Docker (persistant, géré par Docker)
  • Bind Mounts (lien direct avec le système hôte)
  • Tmpfs Mounts (stockage temporaire en mémoire)

Persistance des Données

Importance de la Persistance

  • Conservation des données après l'arrêt du conteneur
  • Partage de données entre conteneurs
  • Sauvegarde et restauration facilitées
  • Migration des données entre environnements

Cas d'Usage

  • Bases de données
  • Fichiers de configuration
  • Logs d'application
  • Contenu statique

Choix du Type de Stockage

Critères de Choix

  • Durée de vie des données
  • Performance requise
  • Portabilité des données
  • Besoins en sécurité

Tableau Comparatif

Type Persistance Performance Portabilité
Volumes Oui Excellente Bonne
Bind Mounts Oui Bonne Limitée
Tmpfs Non Excellente Aucune

Points Clés

  • Système de Fichiers : Docker utilise un système en couches pour optimiser l'utilisation du stockage.
  • Types de Stockage : Volumes, Bind Mounts et Tmpfs offrent différentes solutions selon les besoins.
  • Persistance : Essentielle pour certaines applications, avec différents niveaux selon le type de stockage.
  • Choix : La sélection du type de stockage dépend des besoins spécifiques de l'application.

Les Volumes Docker

Objectifs Pédagogiques

Dans cette section, vous allez apprendre à :

  • Comprendre le concept des volumes Docker
  • Créer et gérer des volumes
  • Partager des données entre conteneurs
  • Sauvegarder et restaurer des volumes

Qu'est-ce qu'un Volume Docker ?

Définition

Un volume Docker est un mécanisme de stockage persistant géré par Docker qui permet de :

  • Stocker des données en dehors du système de fichiers du conteneur
  • Persister les données même après la suppression du conteneur
  • Partager des données entre plusieurs conteneurs
  • Gérer le stockage de manière indépendante du cycle de vie des conteneurs

Avantages des Volumes

  • Meilleure performance que les bind mounts
  • Fonctionne sur tous les systèmes d'exploitation
  • Peut être géré via l'API Docker
  • Plus sécurisé pour le partage entre conteneurs

Gestion des Volumes

Commandes de Base

# Créer un volume
docker volume create mon_volume

# Lister les volumes
docker volume ls

# Inspecter un volume
docker volume inspect mon_volume

# Supprimer un volume
docker volume rm mon_volume

# Nettoyer les volumes inutilisés
docker volume prune

Utilisation avec les Conteneurs

# Monter un volume lors du lancement d'un conteneur
docker run -v mon_volume:/data nginx

# Créer et monter un volume anonyme
docker run -v /data nginx

# Monter un volume en lecture seule
docker run -v mon_volume:/data:ro nginx

Partage de Données

Entre Conteneurs

# Créer un conteneur avec un volume
docker run -d --name conteneur1 -v mon_volume:/data nginx

# Utiliser le même volume dans un autre conteneur
docker run -d --name conteneur2 -v mon_volume:/data nginx

# Utiliser les volumes d'un autre conteneur
docker run --volumes-from conteneur1 nginx

Bonnes Pratiques

  • Nommer explicitement les volumes pour une meilleure traçabilité
  • Utiliser des étiquettes pour organiser les volumes
  • Nettoyer régulièrement les volumes inutilisés
  • Documenter l'utilisation des volumes dans le projet

Sauvegarde et Restauration

Sauvegarder un Volume

# Créer une sauvegarde
docker run --rm -v mon_volume:/source -v $(pwd):/backup alpine tar -czf /backup/mon_volume.tar.gz -C /source .

Restaurer un Volume

# Restaurer depuis une sauvegarde
docker run --rm -v mon_volume:/target -v $(pwd):/backup alpine sh -c "cd /target && tar -xzf /backup/mon_volume.tar.gz"

Points Clés

  • Volumes : Solution de stockage persistant native de Docker, offrant les meilleures performances.
  • Gestion : Commandes simples pour créer, lister, inspecter et supprimer les volumes.
  • Partage : Facilité de partage des données entre conteneurs avec une bonne isolation.
  • Sauvegarde : Possibilité de sauvegarder et restaurer les données des volumes.

Les Bind Mounts

Objectifs Pédagogiques

Dans cette section, vous allez apprendre à :

  • Comprendre le concept des bind mounts
  • Utiliser les bind mounts avec Docker
  • Gérer les permissions et la sécurité
  • Appliquer les bonnes pratiques

Introduction aux Bind Mounts

Définition

Un bind mount est un mécanisme qui permet de monter directement un fichier ou un répertoire du système hôte dans un conteneur. Contrairement aux volumes Docker, les bind mounts :

  • Dépendent de la structure de fichiers de l'hôte
  • Peuvent être modifiés directement depuis l'hôte
  • Offrent un accès direct aux fichiers du système
  • Sont parfaits pour le développement

Cas d'Usage

  • Développement d'applications
  • Configuration des conteneurs
  • Partage de fichiers avec l'hôte
  • Tests et débogage

Utilisation des Bind Mounts

Syntaxe de Base

# Syntaxe longue (recommandée)
docker run --mount type=bind,source=/chemin/hote,target=/chemin/conteneur nginx

# Syntaxe courte (ancienne)
docker run -v /chemin/hote:/chemin/conteneur nginx

# Monter en lecture seule
docker run --mount type=bind,source=/chemin/hote,target=/chemin/conteneur,readonly nginx

Exemples Pratiques

# Monter un répertoire de développement
docker run -d --name dev-env \
  --mount type=bind,source=$(pwd),target=/app \
  node:alpine

# Monter un fichier de configuration
docker run -d --name web-server \
  --mount type=bind,source=/etc/nginx/nginx.conf,target=/etc/nginx/nginx.conf,readonly \
  nginx

Gestion des Permissions

Permissions et Propriété

  • Les permissions sont héritées du système hôte
  • L'utilisateur dans le conteneur doit avoir les droits appropriés
  • Attention aux différences de UID/GID entre l'hôte et le conteneur

Exemple avec Permissions

# Définir l'utilisateur pour le conteneur
docker run -d --name app \
  --mount type=bind,source=$(pwd),target=/app \
  --user "$(id -u):$(id -g)" \
  node:alpine

Bonnes Pratiques

Recommandations

  • Utiliser la syntaxe --mount pour plus de clarté
  • Toujours spécifier les chemins absolus
  • Préférer le mode lecture seule quand possible
  • Gérer correctement les permissions
  • Documenter l'utilisation des bind mounts

Sécurité

  • Limiter l'accès aux fichiers sensibles
  • Éviter les bind mounts en production
  • Vérifier les permissions avant le montage
  • Utiliser SELinux ou AppArmor si disponible

Points Clés

  • Bind Mounts : Permettent de monter directement des fichiers de l'hôte dans un conteneur.
  • Utilisation : Parfaits pour le développement et la configuration des conteneurs.
  • Permissions : Nécessitent une attention particulière à la gestion des droits d'accès.
  • Sécurité : À utiliser avec précaution, particulièrement en environnement de production.

Les Montages tmpfs

Objectifs Pédagogiques

Dans cette section, vous allez apprendre à :

  • Comprendre le concept des tmpfs mounts
  • Utiliser le stockage temporaire en mémoire
  • Gérer les données sensibles
  • Optimiser les performances avec tmpfs

Introduction aux Tmpfs Mounts

Définition

Les tmpfs mounts sont des montages temporaires stockés en mémoire vive (RAM) qui permettent de :

  • Stocker des données temporaires à haute vitesse
  • Gérer des informations sensibles de manière sécurisée
  • Améliorer les performances pour les données fréquemment accédées
  • Effacer automatiquement les données à l'arrêt du conteneur

Utilisation des Tmpfs Mounts

Syntaxe de Base

# Créer un montage tmpfs
docker run -d --name app --tmpfs /app/temp nginx

# Spécifier la taille et les permissions
docker run -d --name app --tmpfs /app/temp:rw,size=100M,mode=700 nginx

# Utiliser la syntaxe mount
docker run -d --name app \
  --mount type=tmpfs,destination=/app/temp,tmpfs-size=100M,tmpfs-mode=700 \
  nginx

Cas d'Usage

Applications Courantes

  • Stockage de sessions temporaires
  • Cache d'application
  • Fichiers de travail temporaires
  • Données sensibles (clés, tokens)

Exemple Pratique

# Application web avec cache en mémoire
docker run -d --name webapp \
  --tmpfs /app/cache:rw,size=100M \
  --tmpfs /app/sessions:rw,size=50M,mode=700 \
  mon-app:latest

Considérations

Avantages

  • Performances élevées (stockage en RAM)
  • Sécurité accrue pour les données sensibles
  • Nettoyage automatique à l'arrêt du conteneur
  • Pas d'impact sur le système de fichiers

Limitations

  • Données non persistantes
  • Utilisation de la mémoire système
  • Taille limitée par la RAM disponible
  • Non partageable entre conteneurs

Points Clés

  • Tmpfs : Solution de stockage temporaire en mémoire pour les données volatiles.
  • Performance : Excellentes performances grâce au stockage en RAM.
  • Sécurité : Idéal pour les données sensibles nécessitant un effacement automatique.
  • Usage : Parfait pour le cache et les fichiers temporaires.

Travail Pratique

Objectifs du TP

Dans ce travail pratique, vous allez :

  • Mettre en place une application web avec différents types de stockage
  • Configurer les volumes pour les données persistantes
  • Utiliser les bind mounts pour le développement
  • Implémenter le stockage temporaire avec tmpfs

Préparation de l'Environnement

Structure du Projet

# Création de la structure du projet
mkdir app-demo
cd app-demo

# Création des répertoires
mkdir src config data logs
touch src/index.js config/app.conf

Configuration des Volumes

Création des Volumes

# Création des volumes pour les données persistantes
docker volume create app_data
docker volume create app_logs

# Vérification des volumes créés
docker volume ls
docker volume inspect app_data

Mise en Place de l'Application

Dockerfile

FROM node:alpine

WORKDIR /app

COPY package*.json ./
RUN npm install

COPY src/ ./src/

VOLUME ["/app/data", "/app/logs"]
EXPOSE 3000

CMD ["node", "src/index.js"]

Lancement avec Différents Types de Stockage

# Lancement en mode développement avec bind mounts
docker run -d --name app-dev \
  -v $(pwd)/src:/app/src \
  -v $(pwd)/config:/app/config \
  --mount type=volume,source=app_data,target=/app/data \
  --mount type=volume,source=app_logs,target=/app/logs \
  --tmpfs /app/temp:rw,size=100M \
  -p 3000:3000 \
  app-demo

Tests et Vérifications

Vérification du Stockage

# Vérification des montages
docker inspect app-dev

# Test d'écriture dans les différents types de stockage
docker exec app-dev sh -c "echo 'test' > /app/data/test.txt"
docker exec app-dev sh -c "echo 'test' > /app/temp/test.txt"

# Vérification de la persistance
docker stop app-dev
docker start app-dev
docker exec app-dev ls -la /app/data /app/temp
